## 二、3200 端口现状审查

### 2.1 鉴权机制：已有但不完整

**已有的部分：**

- `requirePermission()` 函数存在，写接口理论上需要登录态
- 会话通过 `sessionStore`（内存 Map）管理，TTL 12 小时
- 角色权限分 5 级：`viewer / editor / publisher / admin / owner`
- 密码使用 `sha256("cfqh-admin:" + password)` 散列

**实际存在的问题：**

**问题 1：本地模式（LocalStore）下鉴权逻辑短路**

`getAuthState()` 在非 cloud 模式下直接返回：

```js
// server.cjs ~520行
if (!cloudReady) {
  return {
    authenticated: false,
    bootstrapRequired: false,
    ...
  };
}
```

这意味着当 `.env.admin-web` 未配置云凭据时，服务以本地模式运行，此时写接口的鉴权行为需要逐一确认是否真正拦截了未登录请求。本地模式是开发常态，如果写接口在本地模式下不校验登录态，则本地运行时任何人都可以直接调用写接口。

**问题 2：密码散列无真正随机盐**

当前散列方式：`sha256("cfqh-admin:" + password)`

固定前缀 `cfqh-admin:` 不是随机盐，属于"加固定前缀"而非"加盐"。这意味着：
- 相同密码的两个账号，散列值完全相同
- 一旦散列值泄露，可以用彩虹表或预计算方式反推

建议改为每个账号存储独立随机盐，散列时使用 `sha256(salt + password)` 或直接使用 `bcrypt`。

**问题 3：会话存储在内存，服务重启后全部失效**

`sessionStore = new Map()` 是进程内存，服务重启后所有已登录用户的会话立即失效，需要重新登录。这在开发环境可以接受，但如果 3200 服务被当作生产工具使用，会造成频繁掉登录。

**问题 4：CORS 完全开放**

```js
// server.cjs ~663行
'Access-Control-Allow-Origin': '*'
```

配合监听地址 `0.0.0.0`（局域网可访问），任何局域网内的页面都可以向 3200 发起跨域请求。如果后台被当作生产工具，建议将 CORS 收束到明确的来源。

**问题 5：readBody 无请求体大小限制**

```js
// server.cjs ~691行
req.on('data', (chunk) => { raw += chunk; });
```

没有对请求体大小做限制，理论上可以发送超大请求体导致内存占用异常。CSV 导入接口尤其需要注意这一点。

---

### 2.2 冲突保护：基础已有，但未确认全面覆盖

**已有的部分：**

`createPageConflictError()` 函数已存在（server.cjs ~777行），说明冲突检测逻辑已经被设计进来：

```js
function createPageConflictError(pageKey, payload, sourceMode) {
  const error = new Error('当前页面已被其他老师更新，请先重新拉取后再保存。');
  error.statusCode = 409;
  error.code = 'PAGE_CONFLICT';
  ...
}
```

`buildPageRevision()` 也已存在，基于 `_updatedAt` 时间戳生成 revision 字符串。

**需要确认的问题：**

- 页面保存接口（`PUT /api/page/{pageKey}`）是否真正在保存前比对了客户端提交的 revision 与服务端当前 revision？
- 集合条目的 CRUD 接口是否也有类似保护？
- 前端（admin-web/app.js）在加载页面数据时是否保存了 revision，并在提交时带上？

如果 `createPageConflictError` 只是定义了但没有被调用，则冲突保护实际上是空的。

---

### 2.3 数据源：三层并存，职责边界模糊

当前 3200 服务存在三种运行模式：

| 模式 | 触发条件 | 写操作目标 |
|------|----------|------------|
| `cloud` | 配置了 `CLOUDBASE_APIKEY` 或腾讯云密钥 | 云数据库 |
| `local` | 未配置任何云凭据 | `database/local-admin-data.json` |
| `cloud-required` | 有 `CLOUD_ENV_ID` 但无凭据 | 只读降级 |

**核心问题：**

本地模式写入的是 `local-admin-data.json`，这份数据**不会自动同步到云端**。如果开发者在本地模式下修改了内容，切换到云模式后，本地的修改不会出现在云端，也不会有任何提示。

同时，`/api/health` 接口会返回当前模式，但后台 UI 是否在显眼位置展示了"当前是本地模式还是云端模式"，需要确认。如果没有，运营人员很容易在本地模式下修改内容，误以为已经同步到生产环境。

## 三、前后端数据传输链路审查

### 3.1 数据流全链路

小程序前端获取内容的完整链路：

```
小程序页面
  → getPublicContent(pageKey)
    → 1. 读 Storage 缓存（TTL 5分钟）
      → 命中：返回缓存 + 后台静默刷新
      → 未命中：
        → 2. 开发环境：请求 http://127.0.0.1:3200/api/public/{pageKey}（超时1500ms）
          → 成功：写缓存，返回
          → 失败：
        → 3. 调用云函数 publicContent（最多重试3次，300ms延迟，指数退避）
          → 成功：写缓存，返回
          → 失败（CLOUD_REQUIRE_REMOTE=true）：抛出错误
          → 失败（CLOUD_REQUIRE_REMOTE=false）：
        → 4. 使用 fallback 静态数据
```

### 3.2 具体问题

**问题 1：缓存版本号硬编码，强制刷新需要改代码**

```ts
// content.ts 第10行
const CACHE_VERSION = '2026-03-21-materials-packages-v1';
```

每次需要强制让所有用户刷新缓存时，必须修改这个字符串并重新发布小程序。这意味着：
- 后台修改了内容，但用户的缓存还没过期（最多5分钟），用户看到的是旧内容
- 如果需要立即生效，只能等缓存自然过期，或者发布新版本小程序

**建议：** 将 revision 信息纳入缓存有效性判断。后台保存内容时更新 revision，小程序在后台静默刷新时比对 revision，如果 revision 变化则立即更新缓存。

**问题 2：静默后台刷新失败无感知**

```ts
// content.ts ~186行
refreshContentInBackground(pageKey).catch(() => {
  // 静默失败，不影响用户体验
});
```

静默刷新失败时完全静默，用户和开发者都不知道刷新是否成功。如果云函数持续失败，用户会一直看到过期缓存，直到缓存 TTL 到期后才会触发有感知的加载。

**问题 3：本地预览超时 1500ms 可能过短**

```ts
// content.ts ~159行
timeout: 1500
```

在某些情况下（如 3200 服务刚启动、云数据库响应慢），1500ms 可能不够，导致本地预览频繁失败并降级到云函数。这会让开发者误以为本地预览不工作。

**问题 4：数据归一化层只在服务端，前端无对应处理**

server.cjs 中有大量 `normalize*` 函数（`normalizeHomeHero`、`normalizeCoursesPage` 等），用于处理历史数据兼容。这些函数只在 3200 服务端运行。

当小程序直接调用云函数 `publicContent` 时，云函数返回的是原始云数据库数据，**不经过这些归一化函数**。这意味着：

- 通过 3200 预览的数据，和通过云函数获取的数据，可能结构不完全一致
- 历史遗留数据在云函数路径下可能没有被归一化，导致前端展示异常

**问题 5：fallback 数据与云端数据结构可能存在差异**

`normalize*` 函数的存在本身就说明云端数据存在历史版本差异。fallback 数据（`src/data/contentFallback`）是静态写死的，如果云端数据结构发生变化，fallback 数据不会自动跟进。

当云函数失败时，用户看到的 fallback 内容可能与云端内容结构不一致，甚至展示的是已经废弃的旧版文案。

**问题 6：`CLOUD_REQUIRE_REMOTE = true` 但 fallback 路径仍然存在**

```ts
// cloud.ts
CLOUD_REQUIRE_REMOTE = true
```

```ts
// content.ts ~229行
if (CLOUD_REQUIRE_REMOTE) {
  throw new AppError(...);
}
// 下面还有 fallback 路径
```

`CLOUD_REQUIRE_REMOTE = true` 时，云函数失败会抛出错误，不会走 fallback。但 `getFallbackPublicContent` 函数仍然在文件顶部被调用（第180行），只是最终不会被返回。这不是 bug，但逻辑上有些冗余，也容易让人误以为 fallback 在生产环境仍然生效。

## 五、优化建议与优先级实施步骤

### 第一阶段：系统可信度（建议最先处理）

**P1-1：确认并补全本地模式下的写接口鉴权**

- 检查所有写接口（`PUT /api/page`、`POST/PUT/DELETE /api/collection`、`POST /api/import`、`POST /api/seed/reset`）在本地模式下是否真正拦截了未登录请求
- 如果本地模式下允许无鉴权写入，需要明确这是"开发便利"还是"安全漏洞"，并在文档中标注

**P1-2：确认冲突检测是否真正被调用**

- 检查 `PUT /api/page/{pageKey}` 的实现，确认是否在保存前调用了 `createPageConflictError`
- 如果没有，这是投入最小、收益最高的一项修复：只需在保存逻辑中加入 revision 比对

**P1-3：在后台 UI 顶部显示当前数据源模式**

- 至少展示：当前是本地模式还是云端模式
- 本地模式下，在写操作按钮旁加明显提示："当前修改仅保存到本地，不会同步到生产环境"

**P1-4：给 readBody 加请求体大小限制**

- 建议限制在 5MB 以内（CSV 导入可适当放宽到 10MB）
- 防止超大请求体导致内存异常

---

### 第二阶段：数据传输准确性（建议第二批处理）

**P2-1：将归一化逻辑同步到云函数路径**

- 当前 `normalize*` 函数只在 3200 服务端运行
- 建议将核心归一化逻辑提取为独立模块，同时在 `publicContent` 云函数中引用
- 这样无论数据从哪条路径返回，结构都是一致的

**P2-2：将 revision 纳入缓存有效性判断**

- 后台保存内容时，revision 已经在更新
- 小程序端在静默刷新时，比对新旧 revision，如果变化则立即更新缓存并通知页面刷新
- 这样后台修改后，用户下次打开页面（或当前页面后台刷新完成后）就能看到最新内容，不需要等 5 分钟

**P2-3：静默刷新失败时记录可观测日志**

- 当前静默刷新失败完全静默
- 建议至少在 console 中记录失败原因，方便开发时排查
- 生产环境可以考虑上报到云函数日志

**P2-4：本地预览超时适当放宽**

- 将 1500ms 调整为 2500ms 或 3000ms
- 避免 3200 服务启动慢时频繁触发降级

---

### 第三阶段：运营效率与安全加固（建议第三批处理）

**P3-1：密码散列改为真正加盐**

- 为每个账号生成独立随机盐（存储在 `admin_users` 集合中）
- 散列时使用 `sha256(salt + password)` 或迁移到 `bcrypt`
- 这项改动需要迁移现有账号的密码散列

**P3-2：CORS 收束**

- 将 `Access-Control-Allow-Origin: *` 改为只允许明确的来源（如 `http://localhost:3200`）
- 如果 3200 只用于本地开发，可以限制为 `127.0.0.1` 相关来源

**P3-3：会话持久化（可选）**

- 如果 3200 被当作生产工具使用，考虑将会话持久化到文件或 Redis
- 避免服务重启后所有人需要重新登录

**P3-4：fallback 数据与云端结构同步检查机制**

- 建立一个简单的结构校验脚本，定期比对 fallback 数据与云端数据的字段结构
- 每次前端页面改版时，同步检查 fallback 数据是否需要更新
